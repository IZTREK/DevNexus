Dominando el Backend

Enciclopedia técnica sobre Web APIs, Microservicios y Contenedores.

Unidades de Aprendizaje

1. Tecnologías de Web APIs

Las Web APIs (Application Programming Interfaces) son la columna vertebral de la comunicación moderna entre sistemas distribuidos. Permiten que aplicaciones, servicios y plataformas intercambien datos y funcionalidades de manera estructurada y segura a través de Internet. En el mundo actual del desarrollo web, dominar las APIs es esencial para construir aplicaciones escalables, modulares y eficientes.

¿Qué es HTTP?

HTTP (HyperText Transfer Protocol) es el protocolo de comunicación que hace posible la transferencia de información en la World Wide Web. Funciona bajo un modelo cliente-servidor donde el cliente (navegador o aplicación) solicita recursos y el servidor responde con los datos solicitados.

Métodos HTTP Principales:
  • GET: Solicita datos de un recurso específico. Es idempotente (no modifica datos).
  • POST: Envía datos para crear un nuevo recurso. Ejemplo: registro de usuarios.
  • PUT: Actualiza un recurso existente completamente.
  • PATCH: Actualiza parcialmente un recurso.
  • DELETE: Elimina un recurso específico.
Códigos de Estado HTTP:
  • 2xx (Éxito): 200 OK, 201 Created, 204 No Content
  • 3xx (Redirección): 301 Moved Permanently, 304 Not Modified
  • 4xx (Error del cliente): 400 Bad Request, 401 Unauthorized, 404 Not Found
  • 5xx (Error del servidor): 500 Internal Server Error, 503 Service Unavailable
HTTP/1.1 vs HTTP/2 vs HTTP/3:

HTTP/1.1 procesa una solicitud a la vez por conexión (pipelining limitado).
HTTP/2 introduce multiplexing, permitiendo múltiples solicitudes simultáneas sobre una sola conexión TCP, mejorando significativamente el rendimiento.
HTTP/3 utiliza QUIC sobre UDP en lugar de TCP, reduciendo latencia y mejorando la resiliencia ante pérdida de paquetes.

Simple Object Access Protocol (SOAP)

SOAP es un protocolo de mensajería basado en XML que define un conjunto estricto de reglas para el intercambio estructurado de información en entornos distribuidos y descentralizados. A diferencia de REST, SOAP es un protocolo formal con estándares bien definidos.

Características Principales:
  • Independiente del protocolo: Puede funcionar sobre HTTP, SMTP, TCP, etc.
  • Mensajes XML: Estructura estricta con Envelope, Header y Body.
  • WSDL (Web Services Description Language): Describe automáticamente las operaciones disponibles.
  • WS-Security: Estándar integrado para encriptación, firmas digitales y autenticación.
  • Transacciones ACID: Soporte para operaciones críticas que requieren atomicidad.
Cuándo usar SOAP:
  • Sistemas bancarios y financieros que requieren alta seguridad
  • Aplicaciones empresariales legacy con contratos formales
  • Escenarios donde se necesita garantía de entrega de mensajes
  • Servicios que requieren transacciones complejas y manejo de estado
Ejemplo de Mensaje SOAP:
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header>
    <auth>token123</auth>
  </soap:Header>
  <soap:Body>
    <GetUserRequest>
      <userId>456</userId>
    </GetUserRequest>
  </soap:Body>
</soap:Envelope>

Representational State Transfer (REST)

REST no es un protocolo, sino un estilo arquitectónico que aprovecha HTTP de manera natural. Creado por Roy Fielding en 2000, REST se ha convertido en el estándar de facto para APIs web modernas debido a su simplicidad y escalabilidad.

Principios Fundamentales de REST:
  • Arquitectura Cliente-Servidor: Separación clara de responsabilidades
  • Sin Estado (Stateless): Cada petición contiene toda la información necesaria
  • Cacheable: Las respuestas deben indicar si pueden almacenarse en caché
  • Interfaz Uniforme: URIs como identificadores únicos de recursos
  • Sistema en Capas: Cliente no distingue si se conecta directamente al servidor final
  • Código bajo demanda (opcional): El servidor puede enviar código ejecutable
Diseño de URIs RESTful:
GET    /api/users          → Obtener lista de usuarios
GET    /api/users/123      → Obtener usuario específico
POST   /api/users          → Crear nuevo usuario
PUT    /api/users/123      → Actualizar usuario completo
PATCH  /api/users/123      → Actualizar campos específicos
DELETE /api/users/123      → Eliminar usuario
Ventajas de REST:
  • Ligero y fácil de implementar con JSON
  • Compatible con todos los lenguajes de programación
  • Excelente rendimiento y escalabilidad
  • Caché natural gracias a HTTP
  • Ideal para aplicaciones móviles y web modernas
REST vs RESTful:

Una API es RESTful cuando cumple estrictamente con todos los principios REST. Muchas APIs modernas son "REST-like" pero no 100% RESTful, lo cual es aceptable en la mayoría de casos prácticos.

JSON Web Tokens (JWT)

JWT es un estándar abierto (RFC 7519) que define una forma compacta y auto-contenida de transmitir información de manera segura entre partes como un objeto JSON. Los tokens están firmados digitalmente, lo que permite verificar su autenticidad e integridad.

Estructura de un JWT:

Un JWT consta de tres partes separadas por puntos: xxxxx.yyyyy.zzzzz

  • Header: Contiene el tipo de token (JWT) y el algoritmo de firma (HS256, RS256, etc.) 
    { "alg": "HS256", "typ": "JWT" }
  • Payload: Contiene las "claims" (afirmaciones) sobre el usuario y metadatos 
    {
  "sub": "1234567890",
  "name": "Juan Pérez",
  "role": "admin",
  "iat": 1516239022,
  "exp": 1516242622
}
  • Signature: Garantiza que el token no ha sido alterado 
    HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)
Flujo de Autenticación con JWT:
  1. Usuario envía credenciales (username/password)
  2. Servidor valida credenciales y genera un JWT firmado
  3. Cliente almacena el token (localStorage, sessionStorage, cookie httpOnly)
  4. Cliente envía el token en el header Authorization en cada petición
  5. Servidor verifica la firma y extrae la información del usuario
Ventajas de JWT:
  • Stateless: No requiere almacenar sesiones en el servidor
  • Escalable: Funciona perfectamente en arquitecturas distribuidas
  • Auto-contenido: El token lleva toda la información necesaria
  • Cross-domain: Ideal para aplicaciones móviles y SPAs
  • Microservicios: Fácil de compartir entre diferentes servicios
Buenas Prácticas:
  • Usar HTTPS siempre para evitar interceptación
  • Establecer tiempos de expiración cortos (exp claim)
  • Implementar refresh tokens para renovar JWTs
  • No almacenar información sensible en el payload (es visible en Base64)
  • Validar siempre la firma en el backend

2. Arquitectura de Microservicios

Los microservicios representan un cambio de paradigma en el diseño de software empresarial. En lugar de construir aplicaciones monolíticas donde toda la lógica reside en un único código base, los microservicios descomponen las aplicaciones en servicios pequeños, autónomos y especializados que trabajan juntos. Empresas como Netflix, Amazon, Uber y Spotify han adoptado esta arquitectura para lograr agilidad, escalabilidad y resiliencia sin precedentes.

Arquitectura Monolítica

En una aplicación monolítica, todos los componentes (UI, lógica de negocio, acceso a datos) están fuertemente acoplados y desplegados como una sola unidad. Cualquier cambio requiere redesplegar toda la aplicación.

Desventajas del Monolito:

  • Difícil de escalar: solo se puede escalar la aplicación completa
  • Alta dependencia entre módulos: un bug puede afectar todo el sistema
  • Ciclos de despliegue largos y riesgosos
  • Tecnología monolítica: difícil adoptar nuevas herramientas
  • Equipos grandes trabajando en el mismo código base (conflictos)
Arquitectura de Microservicios

Los microservicios dividen la aplicación en servicios independientes, cada uno responsable de una capacidad de negocio específica. Cada servicio puede desarrollarse, desplegarse y escalarse de manera independiente.

Ventajas de Microservicios:

  • Escalabilidad granular: escala solo los servicios que lo necesitan
  • Independencia tecnológica: cada servicio puede usar el stack más apropiado
  • Despliegue continuo: actualizar un servicio no afecta a los demás
  • Resiliencia: fallo aislado (un servicio caído no tira todo el sistema)
  • Equipos autónomos: equipos pequeños dueños de servicios completos
Tabla Comparativa:
Aspecto Monolito Microservicios
Despliegue Todo o nada Servicios independientes
Escalabilidad Vertical (más recursos) Horizontal (más instancias)
Complejidad Simple al inicio Compleja distribución
Base de Datos Una única BD compartida BD por servicio

Comunicación Sincrónica: HTTP/REST y gRPC

REST APIs: Los microservicios se comunican mediante llamadas HTTP estándar. Simple y ampliamente soportado, pero puede generar latencia en cadenas de llamadas largas.

gRPC: Framework de código abierto desarrollado por Google que usa Protocol Buffers (binario) en lugar de JSON. Mucho más rápido y eficiente, ideal para comunicaciones internas entre microservicios.

Comunicación Asincrónica: Message Brokers

Los servicios publican y consumen mensajes a través de un broker de mensajes (RabbitMQ, Apache Kafka, AWS SQS). El emisor no espera respuesta inmediata, lo que desacopla los servicios y mejora la resiliencia.

Ventajas:

  • Desacoplamiento total entre servicios
  • Buffer ante picos de carga
  • Procesamiento en segundo plano
  • Reintento automático de mensajes fallidos
Event-Driven Architecture (Arquitectura Basada en Eventos)

Los microservicios emiten eventos cuando ocurre un cambio de estado. Otros servicios se suscriben a esos eventos y reaccionan. Ejemplo: cuando se crea un pedido, el servicio de inventario, pagos y notificaciones reaccionan automáticamente.

API Gateway

Un API Gateway actúa como punto de entrada único para todos los clientes. Enruta las peticiones al microservicio correspondiente, maneja autenticación, rate limiting, caché y composición de respuestas.

Tecnologías populares: Kong, AWS API Gateway, NGINX, Traefik

Escalabilidad Horizontal Automática

Con microservicios, puedes escalar horizontalmente (agregar más instancias) solo los servicios bajo alta demanda. Por ejemplo, durante Black Friday, el servicio de catálogo puede tener 50 instancias mientras que el de reportes solo 2.

CI/CD (Continuous Integration / Continuous Deployment)

Cada microservicio tiene su propio pipeline de CI/CD. Los cambios pasan automáticamente por: pruebas unitarias → pruebas de integración → despliegue en staging → despliegue en producción.

Herramientas comunes: Jenkins, GitLab CI, GitHub Actions, CircleCI, ArgoCD

Blue-Green Deployment y Canary Releases

Blue-Green: Mantener dos entornos idénticos (azul y verde). Desplegar en uno mientras el otro atiende tráfico, luego intercambiar. Rollback instantáneo si algo falla.

Canary: Desplegar la nueva versión gradualmente a un pequeño porcentaje de usuarios (5%), monitorear métricas, y si todo va bien, ir incrementando hasta el 100%.

Service Mesh

Una malla de servicios (Service Mesh) como Istio o Linkerd maneja automáticamente: balanceo de carga, cifrado TLS entre servicios, observabilidad (métricas, logs, trazas), circuit breakers y reintentos.

Netflix: Pioneros en Microservicios a Gran Escala

Netflix migró de un monolito a más de 700 microservicios. Manejan millones de peticiones por segundo. Cada servicio es responsable de una capacidad específica: recomendaciones, reproducción, autenticación, facturación, etc.

Tecnologías clave: Spring Boot, Eureka (service discovery), Hystrix (circuit breaker), AWS

Amazon: Todo es un Servicio

Amazon fue de los primeros en adoptar microservicios (2002). El mandato de Jeff Bezos: "Todos los equipos deben exponer sus datos y funcionalidades a través de APIs". Esto permitió su transformación en AWS (Amazon Web Services).

Uber: Arquitectura de Hiper-Escala

Uber tiene más de 2,200 microservicios críticos. Servicios para: cálculo de tarifas, emparejamiento conductor-pasajero, rutas en tiempo real, pagos, promociones, etc. Manejan 15 millones de viajes por día.

Spotify: Equipos Autónomos (Squads)

Spotify organiza sus equipos en "squads" autónomos, cada uno responsable de un microservicio completo. Esto les permite lanzar actualizaciones múltiples veces al día sin interferir entre equipos.

Lecciones Aprendidas:
  • Empezar pequeño: no migrar todo de golpe
  • Invertir en observabilidad desde el día 1
  • Automatizar todo lo posible (testing, deployment)
  • Cultura DevOps: equipos dueños de todo el ciclo de vida
  • No es para todos: evaluar si realmente lo necesitas

3. Contenedores y Orquestación

Los contenedores han revolucionado la forma en que desarrollamos, distribuimos y ejecutamos aplicaciones. Ofrecen un entorno aislado, portable y consistente que garantiza que el software funcione exactamente igual en desarrollo, staging y producción. Docker democratizó la tecnología de contenedores, mientras que Kubernetes se convirtió en el estándar para orquestarlos a gran escala.

¿Qué es Docker?

Docker es una plataforma de código abierto que automatiza el despliegue de aplicaciones dentro de contenedores de software. Un contenedor empaqueta una aplicación junto con todas sus dependencias, bibliotecas y configuraciones necesarias.

Contenedores vs Máquinas Virtuales
Característica Máquina Virtual Contenedor Docker
Sistema Operativo SO completo por VM Comparte Kernel del host
Tamaño GB (varios gigas) MB (megabytes)
Inicio Minutos Segundos
Rendimiento Overhead significativo Near-native (casi nativo)
Aislamiento Completo (hardware virtual) A nivel de proceso
Componentes Principales de Docker:
  • Docker Engine: Motor que ejecuta los contenedores
  • Dockerfile: Archivo de texto con instrucciones para construir una imagen
  • Docker Image: Plantilla inmutable (solo lectura) con la aplicación y sus dependencias
  • Docker Container: Instancia en ejecución de una imagen
  • Docker Hub: Registro público de imágenes (como GitHub pero para contenedores)
  • Docker Compose: Herramienta para definir aplicaciones multi-contenedor
Ejemplo de Dockerfile:
# Imagen base
FROM node:18-alpine

# Directorio de trabajo
WORKDIR /app

# Copiar archivos de dependencias
COPY package*.json ./

# Instalar dependencias
RUN npm install

# Copiar código fuente
COPY . .

# Exponer puerto
EXPOSE 3000

# Comando de inicio
CMD ["npm", "start"]
Comandos Docker Básicos:
docker build -t mi-app .              # Construir imagen
docker run -p 3000:3000 mi-app         # Ejecutar contenedor
docker ps                              # Listar contenedores activos
docker images                          # Listar imágenes
docker stop <container-id>            # Detener contenedor
docker logs <container-id>            # Ver logs
Beneficios de Docker:
  • "Funciona en mi máquina": Elimina este problema para siempre
  • Portabilidad: La misma imagen corre en cualquier lugar
  • Eficiencia: Densidad muy alta (muchos contenedores por servidor)
  • Versionado: Imágenes versionadas como código
  • Rollback fácil: Volver a una versión anterior en segundos

¿Qué es Kubernetes?

Kubernetes (K8s) es un sistema de orquestación de contenedores de código abierto creado por Google. Automatiza el despliegue, escalado y gestión de aplicaciones contenerizadas. Es el estándar de facto en la industria, soportado por todos los proveedores cloud: AWS (EKS), Azure (AKS), Google Cloud (GKE).

Problemas que Resuelve Kubernetes:
  • Auto-escalado: Incrementa o reduce réplicas según la carga
  • Auto-recuperación: Reinicia contenedores que fallan automáticamente
  • Balanceo de carga: Distribuye tráfico entre las réplicas
  • Despliegues sin downtime: Rolling updates y canary deployments
  • Gestión de configuración: ConfigMaps y Secrets
  • Almacenamiento persistente: Manejo de volúmenes
Arquitectura de Kubernetes:

Control Plane (Plano de Control):

  • API Server: Punto de entrada para todas las operaciones
  • etcd: Base de datos clave-valor distribuida (estado del cluster)
  • Scheduler: Decide en qué nodo ejecutar cada Pod
  • Controller Manager: Ejecuta controladores (replicas, endpoints, etc.)

Nodos Worker:

  • Kubelet: Agente que garantiza que los contenedores estén corriendo
  • Kube-proxy: Maneja las reglas de red
  • Container Runtime: Docker, containerd o CRI-O
Conceptos Fundamentales:
  • Pod: Unidad mínima desplegable (uno o más contenedores)
  • Deployment: Define el estado deseado de los Pods (cuántas replicas, imagen, etc.)
  • Service: Abstracción que expone una aplicación (ClusterIP, NodePort, LoadBalancer)
  • Ingress: Enrutamiento HTTP/HTTPS desde fuera del cluster
  • Namespace: Aislamiento lógico de recursos
  • ConfigMap: Configuración externalizada
  • Secret: Datos sensibles (contraseñas, tokens)
Ejemplo de Deployment YAML:
apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: mi-app
        image: mi-app:v1.0
        ports:
        - containerPort: 3000
¿Cuándo usar Kubernetes?
  • Aplicaciones con múltiples microservicios
  • Necesitas alta disponibilidad y escalabilidad
  • Despliegues frecuentes (CI/CD)
  • Aplicaciones cloud-native
  • Multi-nube o híbrido (on-premise + cloud)

Niveles de Virtualización

La virtualización tradicional (VMware, VirtualBox, Hyper-V) emula hardware completo. Cada máquina virtual corre su propio sistema operativo y kernel sobre un hipervisor.

Los contenedores virtualizan a nivel de sistema operativo. Múltiples contenedores comparten el mismo kernel del host, pero están aislados mediante namespaces y cgroups de Linux.

Namespaces de Linux:
  • PID namespace: Aísla los IDs de procesos
  • Network namespace: Interfaz de red aislada
  • Mount namespace: Sistema de archivos independiente
  • UTS namespace: Hostname y dominio propios
  • User namespace: Usuarios y grupos aislados
Control Groups (cgroups):

Limitan y controlan los recursos que usa un contenedor: CPU, memoria, I/O de disco y red. Por ejemplo, puedes limitar un contenedor a usar máximo 2 CPUs y 512MB de RAM.

¿Cuándo usar VMs?
  • Necesitas aislamiento total a nivel de kernel
  • Correr diferentes sistemas operativos (Windows en host Linux)
  • Aplicaciones legacy que requieren SO completo
¿Cuándo usar Contenedores?
  • Aplicaciones modernas (microservicios, cloud-native)
  • Necesitas portabilidad y eficiencia
  • CI/CD y despliegues rápidos
  • Escalado dinámico
El Mejor de Ambos Mundos:

Muchas empresas usan contenedores dentro de VMs. Por ejemplo, en AWS, corres Kubernetes en EC2 (VMs), obteniendo el aislamiento de VMs con la eficiencia de contenedores.

Integración Continua (CI)

Los desarrolladores integran código frecuentemente (varias veces al día) en un repositorio compartido. Cada integración dispara un proceso automatizado que:

  1. Clona el repositorio
  2. Ejecuta pruebas unitarias y de integración
  3. Analiza calidad de código (linting, SonarQube)
  4. Construye la imagen Docker
  5. Sube la imagen a un registro (Docker Hub, AWS ECR)
Despliegue Continuo (CD)

Cada cambio que pasa las pruebas se despliega automáticamente a producción (o staging primero). Con contenedores, el proceso es:

  1. Pipeline de CI construye nueva imagen: mi-app:v1.2.3
  2. Actualiza manifiesto de Kubernetes con la nueva imagen
  3. Kubernetes hace un rolling update (actualización gradual)
  4. Healthchecks verifican que los nuevos Pods funcionen
  5. Tráfico se redirige gradualmente a la nueva versión
  6. Si falla, rollback automático a la versión anterior
Pipeline CI/CD Completo con GitHub Actions:
name: CI/CD Pipeline
on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Run Tests
        run: npm test
      
      - name: Build Docker Image
        run: docker build -t mi-app:${{ github.sha }} .
      
      - name: Push to Registry
        run: docker push mi-app:${{ github.sha }}
      
      - name: Deploy to Kubernetes
        run: |
          kubectl set image deployment/mi-app \\
            mi-app=mi-app:${{ github.sha }}
          kubectl rollout status deployment/mi-app
GitOps con ArgoCD o Flux:

GitOps usa Git como fuente de verdad para la infraestructura y aplicaciones. ArgoCD monitorea constantemente el repositorio Git y sincroniza automáticamente el estado del cluster Kubernetes con lo definido en Git. Cualquier deriva se corrige automáticamente.

Beneficios del CI/CD con Contenedores:
  • Despliegues múltiples veces al día sin miedo
  • Feedback inmediato sobre errores
  • Consistencia total entre entornos
  • Rollback en segundos si algo falla
  • Trazabilidad completa (cada cambio está en Git)

4. Desarrollo de una Web API

Construir una Web API desde cero requiere comprender no solo los conceptos teóricos, sino también las mejores prácticas de implementación. Una API bien diseñada es la diferencia entre un sistema escalable y mantenible versus uno frágil y problemático. En esta sección exploramos el ciclo completo de desarrollo: desde el diseño de endpoints hasta la documentación y el versionado, cubriendo frameworks modernos, patrones de arquitectura y estrategias de testing que utilizan los equipos profesionales.

Principios de Diseño RESTful

Un buen diseño de API empieza con la modelación de recursos. En REST, todo es un recurso identificado por una URI. Los recursos deben ser sustantivos (no verbos) y representar entidades del dominio de negocio.

Convenciones de Nomenclatura:
  • Usar sustantivos plurales: /api/users, /api/products
  • Jerarquías claras: /api/users/123/orders (órdenes del usuario 123)
  • Minúsculas y guiones: /api/product-categories
  • Evitar verbos: ❌ /api/getUsers ✅ GET /api/users
  • Versionado en la URL: /api/v1/users, /api/v2/users
Estructura de Respuestas Consistente:
{
  "success": true,
  "data": {
    "id": 123,
    "name": "Juan Pérez",
    "email": "juan@example.com"
  },
  "meta": {
    "timestamp": "2026-03-04T10:30:00Z",
    "version": "v1"
  }
}
Manejo de Errores Estandarizado:
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email inválido",
    "details": [
      {
        "field": "email",
        "issue": "Formato de email incorrecto"
      }
    ]
  },
  "meta": {
    "timestamp": "2026-03-04T10:30:00Z"
  }
}
Paginación, Filtrado y Ordenamiento:
GET /api/users?page=2&limit=20&sort=-createdAt&filter=active
GET /api/products?category=electronics&minPrice=100&maxPrice=500

Response:
{
  "success": true,
  "data": [...],
  "pagination": {
    "page": 2,
    "limit": 20,
    "total": 150,
    "pages": 8
  }
}
HATEOAS (Hypermedia as the Engine of Application State):

Incluir enlaces hipermedia en las respuestas permite que los clientes descubran dinámicamente las acciones disponibles: 

{
  "id": 123,
  "name": "Juan Pérez",
  "links": [
    { "rel": "self", "href": "/api/users/123" },
    { "rel": "orders", "href": "/api/users/123/orders" },
    { "rel": "update", "href": "/api/users/123", "method": "PUT" },
    { "rel": "delete", "href": "/api/users/123", "method": "DELETE" }
  ]
}

Node.js con Express.js

Express es el framework web minimalista más popular para Node.js. Perfecto para APIs rápidas y ligeras. 

const express = require('express');
const app = express();

app.use(express.json());

// Middleware de autenticación
app.use('/api', authMiddleware);

// Rutas
app.get('/api/users', async (req, res) => {
  try {
    const users = await User.find();
    res.json({ success: true, data: users });
  } catch (error) {
    res.status(500).json({ 
      success: false, 
      error: { message: error.message } 
    });
  }
});

app.listen(3000, () => console.log('API running on port 3000'));
Python con FastAPI

FastAPI es un framework moderno de alto rendimiento para construir APIs con Python 3.7+. Generación automática de documentación OpenAPI, validación de tipos con Pydantic, y async/await nativo. 

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    name: str
    email: str
    age: int

@app.get("/api/users")
async def get_users():
    users = await db.fetch_users()
    return {"success": True, "data": users}

@app.post("/api/users")
async def create_user(user: User):
    if user.age < 18:
        raise HTTPException(status_code=400, detail="Debe ser mayor de edad")
    await db.insert_user(user)

const jwt = require('jsonwebtoken');

// Generar token al login
app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;
  const user = await User.findOne({ email });
  
  if (!user || !await bcrypt.compare(password, user.password)) {
    return res.status(401).json({ 
      success: false, 
      error: { message: 'Credenciales inválidas' } 
    });
  }
  
  const token = jwt.sign(
    { userId: user._id, role: user.role },
    process.env.JWT_SECRET,
    { expiresIn: '24h' }
  );
  
  res.json({ success: true, data: { token, user } });
});

// Middleware de verificación
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  
  if (!token) {
    return res.status(401).json({ 
      success: false, 
      error: { message: 'Token requerido' } 
    });
  }
  
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;
    next();
  } catch (error) {
    res.status(401).json({ 
      success: false, 
      error: { message: 'Token inválido' } 
    });
  }
};

5. Desarrollo de Aplicaciones para la Nube

El cloud computing ha transformado radicalmente cómo construimos, desplegamos y escalamos aplicaciones. Ya no necesitas comprar servidores físicos ni preocuparte por la infraestructura; los proveedores cloud como AWS, Azure y Google Cloud ofrecen recursos bajo demanda, escalado automático y disponibilidad global. Desarrollar para la nube requiere adoptar nuevos patrones arquitectónicos: aplicaciones cloud-native diseñadas para aprovechar la elasticidad, resiliencia y servicios administrados que ofrece la nube.

Amazon Web Services (AWS)

AWS es el líder del mercado con 32% de cuota global. Ofrece más de 200 servicios completamente administrados desde centros de datos en 30 regiones geográficas alrededor del mundo.

Servicios principales:

  • EC2 (Elastic Compute Cloud): Servidores virtuales escalables
  • S3 (Simple Storage Service): Almacenamiento de objetos ilimitado
  • RDS (Relational Database Service): Bases de datos administradas (MySQL, PostgreSQL, etc.)
  • Lambda: Funciones serverless (ejecuta código sin gestionar servidores)
  • ECS/EKS: Orquestación de contenedores (Docker y Kubernetes)
  • API Gateway: Creación y gestión de APIs
  • CloudFront: CDN global para distribución de contenido
Microsoft Azure

Azure es la segunda plataforma más grande (23% del mercado). Integración perfecta con el ecosistema Microsoft (Active Directory, Office 365, .NET). Popular en empresas y gobierno.

Servicios principales:

  • Azure Virtual Machines: VMs en la nube
  • Azure App Service: Hosting de aplicaciones web y APIs
  • Azure Functions: Serverless computing
  • Azure SQL Database: Base de datos SQL administrada
  • Azure Kubernetes Service (AKS): Kubernetes administrado
  • Azure DevOps: CI/CD y gestión de proyectos
Google Cloud Platform (GCP)

GCP (10% del mercado) destaca por sus servicios de Big Data, Machine Learning y Kubernetes (creado por Google). Red global de alta velocidad.

Servicios principales:

  • Compute Engine: Máquinas virtuales
  • Cloud Run: Contenedores serverless
  • BigQuery: Data warehouse para análisis masivo
  • Cloud Functions: Funciones serverless
  • GKE (Google Kubernetes Engine): Kubernetes administrado más avanzado
  • Firebase: Backend-as-a-Service para aplicaciones móviles

12-Factor App: Principios Fundamentales

Metodología creada por Heroku que define 12 mejores prácticas para construir aplicaciones SaaS modernas y cloud-native:

  • I. Codebase: Un código base versionado en Git, múltiples despliegues
  • II. Dependencies: Declarar explícitamente todas las dependencias
  • III. Config: Configuración en variables de entorno, no en código
  • IV. Backing Services: Tratar bases de datos, colas, etc. como recursos adjuntos
  • V. Build, Release, Run: Separar estrictamente las etapas
  • VI. Processes: Ejecutar la app como procesos stateless
  • VII. Port Binding: Exportar servicios vía port binding
  • VIII. Concurrency: Escalar mediante el modelo de procesos
  • IX. Disposability: Inicio rápido y apagado graceful
  • X. Dev/Prod Parity: Entornos lo más similares posible
  • XI. Logs: Logs como streams de eventos
  • XII. Admin Processes: Tareas de administración como procesos one-off
Serverless Computing

Serverless no significa "sin servidores", sino que no gestionas la infraestructura. Escribes funciones que se ejecutan en respuesta a eventos y pagas solo por el tiempo de ejecución (milisegundos).

Ventajas:

  • Sin gestión de servidores ni actualizaciones de SO
  • Escalado automático desde 0 hasta miles de ejecuciones
  • Pago por uso real (no por servidor idle)
  • Alta disponibilidad integrada
Ejemplo de AWS Lambda (Serverless):
// AWS Lambda function en Node.js
exports.handler = async (event) => {
  const userId = event.pathParameters.id;
  
  // Consultar DynamoDB
  const user = await dynamoDB.get({
    TableName: 'Users',
    Key: { userId }
  }).promise();
  
  return {
    statusCode: 200,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ success: true, data: user.Item })
  };
};

// Se ejecuta solo cuando recibe una petición HTTP
// API Gateway → Lambda → DynamoDB
// No pagas cuando no hay requests
